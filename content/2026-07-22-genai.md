Most LangGraph tutorials stop at "define a state, add some nodes, connect some edges." That gets you a working demo. It doesn't prepare you for the questions that come up the moment you try to build something that retries, waits for a human, or splits work across multiple agents. This is a walkthrough of five of those questions — the ones that separate "I made a graph" from "I understand how the graph actually behaves."

## Reducers: Why `Annotated[list, operator.add]` Changes Everything

**Default behavior — overwrite** — When a state field is declared as a plain type in your `TypedDict`, every node that returns a value for that key replaces whatever was there before. If two nodes both write to a `messages` field without a reducer, the second one to run simply wins, and everything the first node wrote is gone.

**With a reducer — merge** — Wrapping the field as `Annotated[list, operator.add]` tells LangGraph how to combine the old value with the new one instead of replacing it. For a list, `operator.add` means concatenation: each node's output gets appended to what's already there rather than overwriting it. This is exactly why message history in agent graphs uses this pattern — every LLM call and tool result needs to accumulate, not clobber the previous turn.

**When to write a custom reducer** — The built-in `operator.add` only covers simple concatenation. You'd reach for a custom reducer function when the merge logic is more specific than "append everything" — deduplicating messages by ID, keeping only the highest-confidence result when two branches write conflicting values, or merging two dictionaries key by key instead of replacing one with the other. A custom reducer is just a function of `(existing_value, new_value) -> merged_value`, and you attach it the same way: `Annotated[YourType, your_merge_function]`.

- No annotation: last write wins, full overwrite.
- `operator.add`: values are concatenated or summed, useful for lists and counters.
- Custom reducer: needed when the merge rule is domain-specific — dedup, conflict resolution, partial updates.

## `add_conditional_edges`: Routing Based on What Actually Happened

**The core idea** — A conditional edge doesn't connect two fixed nodes. It connects a source node to a routing function that inspects the current state and returns the name of whichever node should run next. This is what makes branching possible — the path through the graph depends on the data, not a diagram drawn in advance.

Here's the loop most tool-using agents are built around: an LLM node decides whether to call a tool, a tool node executes it if needed, and control returns to the LLM node to read the result — repeating until the LLM stops requesting tools.

```python
from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolNode

def call_model(state):
    response = llm_with_tools.invoke(state["messages"])
    return {"messages": [response]}

def route_after_model(state):
    last_message = state["messages"][-1]
    if getattr(last_message, "tool_calls", None):
        return "tools"
    return END

builder = StateGraph(AgentState)
builder.add_node("agent", call_model)
builder.add_node("tools", ToolNode(tools))

builder.set_entry_point("agent")
builder.add_conditional_edges("agent", route_after_model, {"tools": "tools", END: END})
builder.add_edge("tools", "agent")

graph = builder.compile()
```

The routing function, `route_after_model`, is the whole trick: it checks whether the last message carries tool calls, and only sends the graph to `END` when it doesn't. The edge from `"tools"` back to `"agent"` is a normal fixed edge — it's the conditional edge coming out of `"agent"` that creates the loop, by deciding each time whether to go around again or stop.

> A conditional edge isn't a fork in a diagram. It's a function that gets to look at the actual state before deciding where the graph goes next.

## What a Checkpointer Actually Does

**Persisting state, not just messages** — A checkpointer saves a snapshot of the entire graph state after each step, keyed by a `thread_id`. Without one, a graph that pauses — for an interrupt, a crash, or a deliberate stop — has nothing to resume from; the next invocation starts from scratch.

**`thread_id` as a persistent cursor** — Reusing the same `thread_id` on your next call resumes exactly where that thread left off, picking up the saved state; using a new one starts a fresh thread with empty state. This is the mechanism that lets a long-running agent be paused for hours and picked back up later, as long as the caller remembers which `thread_id` it was using.

**Checkpoint namespaces and time travel** — Because every step's state gets its own checkpoint under the same thread, you can list a thread's checkpoint history and replay the graph from any earlier point instead of only the latest one — rewinding to a prior decision and forking a different path from there, while the original run stays intact. This is what people mean by "time-travel debugging" in LangGraph: it's not magic, it's just that every intermediate state was saved and is individually addressable.

**MemorySaver vs. a Postgres saver** — `MemorySaver` (also seen as `InMemorySaver`) keeps checkpoints in process memory — fine for local development and testing, gone the moment the process restarts. A Postgres-backed saver persists checkpoints to an actual database, which is what you need the moment more than one process needs to read that state, or the agent needs to survive a restart, or you're running in production where losing an in-progress human-approval workflow to a redeploy isn't acceptable.

## `interrupt()` and `Command(resume=...)`: Pausing Mid-Node

**How it works** — Calling `interrupt(payload)` inside a node pauses graph execution at that exact point and surfaces the payload to whatever called the graph. The caller inspects that payload, decides on a response, and invokes the graph again with `Command(resume=your_response)`. LangGraph passes that resume value back as the return value of the original `interrupt()` call, and the node continues from there — a checkpointer is required, since this only works because the state was persisted at the moment of the pause.

One detail worth knowing: resuming re-executes the node from its start, not from the exact line that called `interrupt()`. Any side effects before the interrupt call in that node will run again on resume, so idempotency matters if the node does more than just call `interrupt()`.

**Static `interrupt_before` / `interrupt_after`** — Set at compile time — `graph.compile(interrupt_before=["tools"])` — this pauses the graph every single time execution reaches that node, unconditionally. It's a blunt instrument: useful when you always want a human to review before a specific node runs, but it can't express "only pause if this particular tool call looks risky."

**When to use each** — Static interrupts fit a workflow where a specific node is always sensitive — reviewing every database write, for instance, regardless of what the write actually contains. `interrupt()` fits the more common real case: pause conditionally, based on what's actually in the state — approve a tool call only if it's a destructive one, or only surface a review step when the model's confidence looks low. Most production human-in-the-loop patterns lean on `interrupt()` precisely because the decision to pause is itself data-dependent.

## Supervisor, Hierarchical, and Network Topologies

**Supervisor** — One agent acts as a router: it receives the task, decides which specialized agent should handle it next, and each specialist reports back to the supervisor rather than talking to each other directly. This keeps control centralized and makes the overall flow easy to reason about, at the cost of the supervisor becoming a bottleneck if it has to mediate every single handoff.

**Hierarchical** — An extension of the supervisor pattern into multiple layers: a top-level supervisor delegates to team-level supervisors, each of which manages its own set of specialist agents. This scales the supervisor idea to larger agent counts by not making one router responsible for every specialist directly — each layer only has to reason about its own subordinates.

**Network** — Agents can hand off directly to any other agent in the group, without a central router deciding every step. This is the most flexible topology and the hardest to keep predictable, since the path through the system emerges from each agent's own routing decisions rather than one place you can point to and say "this is where the flow is controlled."

- Supervisor: centralized routing, easiest to reason about, can bottleneck.
- Hierarchical: supervisor pattern nested across layers, scales to more agents.
- Network: fully peer-to-peer handoffs, most flexible, hardest to predict.

**Subgraphs and state sharing** — A subgraph can be added as a node inside a parent graph, and by default it shares state with the parent through whatever keys the two state schemas have in common — the parent passes its state in, the subgraph reads and writes the overlapping fields, and the result flows back out. Where the two schemas share a key, updates flow through it in both directions the same way any node's return value would update parent state.

**What happens when the keys don't overlap** — If the subgraph's state schema has no keys in common with the parent's, there's nothing for LangGraph to map automatically — you have to write explicit transformation functions on the way in and the way out: one that maps parent state into the shape the subgraph expects, and one that maps the subgraph's output back into the parent's schema. Skipping this isn't a silent failure so much as the subgraph simply never receiving the data you meant to pass it, because there was no shared field for it to travel through.

> A subgraph isn't automatically "inside" the parent's state. It's automatically inside it only where their schemas actually agree — everywhere else, you're responsible for building the bridge.
